# TypeBridge — 需求文档

> 将飞书消息实时注入你当前聚焦的输入框

---

## 一、产品定位

TypeBridge 是一个 macOS 菜单栏应用（基于 Tauri 构建），核心能力是：

**接收飞书机器人消息 → 自动写入用户当前聚焦的输入框**

典型场景：用户在手机上通过飞书向机器人发一段语音（转文字），桌面端自动将文字注入当前正在使用的编辑器、终端、浏览器输入框，实现"语音驱动桌面输入"。

---

## 二、功能需求

### 2.1 启动与配置窗口

- 首次启动弹出配置窗口，包含以下输入项：
  - 飞书应用 `App ID`（仅支持**自建应用**，飞书长连接不支持商店应用）
  - 飞书应用 `App Secret`
  - 一键测试连接按钮，验证凭据有效性及长连接是否建立成功
- 配置数据持久化到本地（`tauri-plugin-store`），二次启动自动复用，无需重新填写
- 连接成功后窗口可关闭，应用退入托盘继续运行

**设置项（同在配置窗口，状态持久化）：**

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| 注入前确认 | **关闭** | 开启后，收到消息时先弹出确认浮层，用户确认后再注入；关闭时直接注入 |

### 2.2 飞书长连接

- 使用飞书开放平台 **WebSocket 长连接**模式接收机器人消息
  - 接口：`https://open.feishu.cn/open-apis/event/v1/im/message/receive_v1`（Event API v2）
  - 长连接端点：通过 `client.registerEventHandler` 注册消息事件
- 支持自动重连（断线后指数退避重试，最大间隔 60s）
- 连接状态实时写入日志

### 2.3 消息注入（macOS Accessibility API）

- 收到飞书消息后，通过 macOS **Accessibility API**（`AXUIElement`）获取当前系统焦点输入框
- 使用 `CGEventPost` 模拟逐键输入将文本写入目标输入框
- 需要用户在「系统设置 → 隐私与安全性 → 辅助功能」中授权 TypeBridge
- 首次使用时主动引导用户完成权限授权（弹窗提示 + 跳转系统设置）
- 注入结果（成功/失败、目标应用名）写入日志

**注入前确认模式（可在设置中切换）：**
- 默认**关闭**，收到消息直接注入
- 开启后，在屏幕角落显示一个轻量确认浮层（悬浮窗），展示消息预览，用户点击「注入」后执行，点击「忽略」丢弃

**图片消息处理：**
- 飞书消息中含有图片时（单独图片或图文混合），Go sidecar 调用飞书 API 下载图片原始字节
- Rust 侧将图片写入 macOS **剪贴板**（`NSPasteboard`，`NSPasteboardTypeTIFF` / `NSPasteboardTypePNG`）
- 写入剪贴板后模拟 `Cmd+V` 触发粘贴，将图片粘入当前焦点输入框
- 若消息同时包含文本和图片：先注入文本，再粘贴图片（顺序保持消息原始顺序）
- 若目标输入框不支持图片粘贴（如纯文本终端），粘贴失败时写入日志，不影响文本注入

**无焦点输入框时的处理：**
- 检测到无可写入的焦点元素时，通过 `UNUserNotificationCenter` 发送系统推送通知，提示用户"当前无聚焦输入框，消息已暂存，请聚焦后重试"
- 暂存最近 1 条消息（文本 + 图片），用户点击通知后触发注入（或仅复制文本到剪贴板）

### 2.4 系统托盘

应用退出窗口后驻留托盘，托盘图标菜单项：

| 菜单项 | 行为 |
|--------|------|
| **连接飞书** | 唤出配置窗口，可重新填写凭据或重新建立连接 |
| **消息日志** | 唤出日志窗口（详见 2.5） |
| **退出应用** | 断开连接并完全退出进程 |

托盘图标状态：
- 已连接：图标正常显示
- 未连接 / 断线：图标灰色或角标红点

### 2.5 日志系统

**日志窗口内容（实时滚动）：**
- 飞书连通性日志（连接成功、断线、重连尝试）
- 收到的消息内容（发送人、时间、消息文本，敏感内容可选脱敏）
- 注入状态（目标应用名、注入成功/失败原因）
- 权限状态变化

**日志文件：**
- 写入路径：`~/Library/Logs/TypeBridge/typebridge-YYYY-MM-DD.log`
- 按天滚动，保留最近 30 天
- 日志窗口底部提供「在访达中显示」按钮，点击后用 `NSWorkspace.selectFile` 在 Finder 中高亮该文件

## 三、技术架构

### 3.1 技术选型

| 层 | 技术 |
|----|------|
| 桌面框架 | Tauri 2.x（Rust + WebView） |
| 前端 UI | React + TypeScript + Tailwind CSS |
| 状态管理 | Zustand |
| 本地存储 | `tauri-plugin-store`（JSON 加密存储凭据） |
| 飞书 SDK | 在 Rust 侧通过 `reqwest` + `tokio-tungstenite` 实现长连接；或调用飞书官方 Go SDK 的 HTTP API 作为 sidecar |
| 文字注入 | Rust 调用 macOS `ApplicationServices.framework`（`AXUIElement`） |
| 系统通知 | Tauri `notification` 插件（封装 `UNUserNotificationCenter`） |
| 日志 | `tracing` + `tracing-appender`（按天滚动） |

```

│   ├── mod.rs

│   ├── mod.rs

└── logger.rs            # 日志初始化与文件路径管理
```

## 五、权限说明

| 权限 | 用途 | 申请时机 |
|------|------|---------|
| 辅助功能（Accessibility） | 检测焦点输入框并注入文字 | 首次尝试注入前 |
| 通知（Notifications） | 无焦点时提示用户 | 首次启动时 |
| 网络（Outbound HTTP/WS） | 连接飞书开放平台、下载图片 | Tauri 默认允许 |

## 六、后续扩展方向（不在 v1 范围）

- 支持更多消息来源（Slack、Telegram、微信等）
- 消息过滤规则（仅转发特定关键词 / 特定发送人）
- 多设备同步（配置云端备份）
- Windows 版本（替换 AXUIElement 为 UIAutomation）
- 消息历史记录与搜索
