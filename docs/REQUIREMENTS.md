# TypeBridge — 需求文档

> 将飞书 / 钉钉 / 企微机器人消息实时注入你当前聚焦的输入框

---

## 一、产品定位

TypeBridge 是一个 macOS 菜单栏应用（基于 Tauri 构建），核心能力是：

**接收主流 IM 平台的机器人消息（飞书 / 钉钉 / 企业微信）→ 自动写入用户当前聚焦的输入框**

典型场景：用户在手机上通过 IM 向机器人发一段语音（转文字），桌面端自动将文字注入当前正在使用的编辑器、终端、浏览器输入框，实现"语音驱动桌面输入"。

**多渠道核心理念**：用户可以同时接入飞书 / 钉钉 / 企微的机器人，三家收到的消息**进入同一个 FIFO 队列**，依次粘到当前焦点输入框。各渠道彼此独立、互不阻塞——用户可以随手在任一 IM 给 bot 发消息，桌面侧串行处理。

---

## 二、功能需求

### 2.1 启动与配置窗口

- 首次启动弹出配置窗口，包含以下输入项：
  - 飞书应用 `App ID`（仅支持**自建应用**，飞书长连接不支持商店应用）
  - 飞书应用 `App Secret`
- 配置数据持久化到本地（`tauri-plugin-store`），二次启动自动复用，无需重新填写
- **关闭窗口即隐藏（不退出）**：点击窗口左上角红色关闭按钮，窗口隐藏，应用进程退入 Dock + 菜单栏托盘继续运行。再次点 Dock / 托盘图标即可秒级唤回，前端 React state 全部保留

**两个按钮（分阶段验证）：**

按钮分两步，对应飞书自建应用长连接接入的真实流程：

| 按钮 | 前置 | 行为 | 成功反馈 | 失败反馈 |
|------|------|------|---------|---------|
| **启动长连接** | App ID / App Secret 都已填写 | 先做表单校验（非空 + App ID 必须以 `cli_` 开头）→ 启动 Go sidecar 建立 WebSocket 长连接 | 状态变"已连接"+ 提示"请前往飞书开发者后台完成长连接验证" | 校验失败时在对应字段下显示红色错误文字；启动失败时在状态下方显示 sidecar stderr 关键行 |
| **测试连接** | 长连接必须处于"已连接" | 通过 Rust→Go stdin 发 `selftest` 命令，Go 侧并行跑 3 个**非破坏性 API probe** 校验消息链路 scope（下载图片 / 发表情 / 回复消息），逐项返回"通过 / 缺失 scope 列表" | 全部通过 → 清单四条全绿（凭据 + 3 项 scope）；否则按条目展示通过/缺失项 | 凭据错误（app_id / app_secret 非法）时清单整体标红，其他情况按失败条目单独提示去飞书开发者后台授权 |

**表单校验规则：**
- App ID 为空 → "App ID 不能为空"
- App ID 格式不对 → "App ID 应以 `cli_` 开头"
- App Secret 为空 → "App Secret 不能为空"
- 两个字段都合法才允许点击「启动长连接」

**连接测试清单显示：**

```
凭据可用（App ID / App Secret 能换到 token）        ✓
──────────────────────────────────
下载图片资源          im:message:readonly           ✓
发表情反应            im:message.reactions:write_only ✓
回复消息              im:message:send_as_bot        ✗ [去授权]
──────────────────────────────────
ⓘ 接收消息事件 需在飞书后台「事件配置」中单独完成
   ① 订阅方式：选择"使用长连接接收事件"并完成验证
   ② 添加事件：搜索 im.message.receive_v1 并勾选提交
   [去事件配置页 ↗]   [查看文档 ↗]
```

事件配置无法通过 API probe 自动校验（无 introspection API、握手不暴露事件列表、唯一可查的 `application.application.get` 需要敏感的 `application:application:self_manage` scope，加进来反而让用户更晕）。所以采用"完整列出步骤 + 直达应用事件配置页"的纯静态引导，不强求自动化。

**scope 校验的"最小粒度/兼容"规则：** 飞书同一个 API 通常存在多个可选 scope（细粒度的新名 + 一个"大包"的旧名 `im:message`）。probe 只关心 API 调用是否返回 `code=99991672`（Access denied，scope 缺失）——用户勾了任意一个匹配 scope 都视为通过，不强制要求用最小粒度。UI 上 `scope_hint` 仅作为"推荐勾选项"展示给用户，失败时给出去开发者后台的深链。

**设置项（同在配置窗口，状态变更立即持久化）：**

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| 输入后自动提交 | **开启** | 开启后，消息写入焦点输入框后模拟按下"提交按键"（默认 Enter），完成一键发送；关闭时仅注入文本不触发提交 |
| 提交按键 | `Enter` | "输入后自动提交"使用的按键。默认 Enter；支持自定义为任意按键或组合键（如 `⌘ + Enter`、`⌃ + Enter` 等）。仅当"输入后自动提交"开启时生效 |

### 2.2 飞书长连接

- 使用飞书开放平台 **WebSocket 长连接**模式接收机器人消息
  - 接口：`https://open.feishu.cn/open-apis/event/v1/im/message/receive_v1`（Event API v2）
  - 长连接端点：通过 `client.registerEventHandler` 注册消息事件
- 支持自动重连（断线后指数退避重试，最大间隔 60s）
- 连接状态实时写入日志

### 2.3 消息输入（剪贴板 + Cmd+V 策略）

- 收到飞书消息后，把内容写入系统剪贴板，然后**模拟 `Cmd+V`** 粘贴到当前前台应用
- 文本和图片走**同一条路径**：文本用 `NSPasteboardTypeString`，图片用 `NSPasteboardTypePNG`
- 粘贴执行完若设置里开启了"输入后自动提交"，追加模拟按下用户配置的提交按键
- 该方案**兼容性最好**——只要目标应用支持标准的"粘贴"行为（几乎所有 macOS 应用），消息就能成功写入。特别是 Electron 应用（VSCode / Slack / Discord / Figma Desktop 等）通过 AX API 查询焦点不稳定，但粘贴完全没问题
- 权限要求：辅助功能（`AXIsProcessTrusted`），因为 `CGEventPost` 发全局按键事件需要该权限

**前台应用保护**（避免字符事件打回 TypeBridge 自己）：
- 粘贴前先通过 `NSWorkspace.frontmostApplication` 查当前前台应用的 bundle identifier
- 若前台是 TypeBridge 本身（`com.typebridge.app`），直接拒绝，`failure_reason = "当前前台是 TypeBridge 自己，请先切换到目标应用"`
- 这比之前用 AX 焦点元素 + role 白名单更可靠——不依赖目标应用是否暴露良好的 AX 树

**剪贴板策略取舍**：
- 粘贴后剪贴板保留消息内容（不回写旧值）。用户若想重新粘贴同一内容直接再按 Cmd+V 即可；若需要原剪贴板请自行保存
- 图文混合消息：优先处理文本（剪贴板 + Cmd+V），再处理图片。图片处理会覆盖剪贴板

**辅助功能权限 —— 仍需授权（但作用已收窄）：**
- 应用启动完成后立刻通过 `AXIsProcessTrusted` 检查权限
- 新的剪贴板 + Cmd+V 策略下，**不再用**AX API 查焦点/读 AXRole；但 `CGEventPost` 发 `Cmd+V` 和自动提交按键给前台应用时，macOS TCC 仍要求应用在"辅助功能"列表里。所以权限检查保留
- **未授予时展示为启动即弹的全屏模态（blocking gate）**：整个主窗口内容被模态覆盖，用户无法继续操作其他 tab。模态包含一段"为什么需要"的说明文字 + 一个大号主按钮「前往授权」，点击后 `open` 直接深链到"系统设置 → 隐私与安全性 → 辅助功能"面板（`x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility`）
- 深链直达辅助功能面板；应用在启动时调 `AXIsProcessTrusted` 已经把自己注册到列表里，用户到了设置页面直接看到 TypeBridge，**无需点"+"手动添加**
- 模态不提供"稍后/跳过"——未授权时应用整体功能不可用（消息一定失败），强制引导授权是"少走一步"而不是"多一个弹窗打扰"
- 授予期间后台每 3s 轮询一次 `check_accessibility`，用户在系统设置里打勾后模态自动消失，无需切换回 TypeBridge 或重启应用
- 未授予时若消息意外到达（比如启动前就已入队）：队列 worker 直接判该条 Failed(reason="辅助功能权限未授予")，飞书侧 CRY 表情 + 文字说明照常发，用户不丢消息可见性
- macOS 会提示一次"TypeBridge 已修改辅助功能设置，是否重启"，按系统提示处理即可；不强制重启

**历史设计（已废弃）**：v0.4 之前尝试过用 `AXUIElement` 查询焦点元素 + AXRole 白名单 + `CGEventPost` 逐字符模拟输入。该方案在 Electron / webview 类应用上 AX API 返回各种 `AXError`（`-25212 NoValue` / `-25204 CannotComplete` 等）导致消息无法写入，已整体替换为本节开头描述的剪贴板 + Cmd+V 方案。相关 AX 查询代码（`get_focused_element` / AXError 映射 / AXRole 白名单）已从运行时移除。

**输入后自动提交（默认开启）：**
- 注入完成后立即模拟按下"提交按键"，完成一键发送。例如在聊天窗口发消息、搜索框搜索都不再需要手动按 Enter
- 默认按键为 `Enter`（macOS `kVK_Return`，keycode 0x24）
- 支持用户自定义：单键（Enter / Tab / Space / 任意字母 / F1–F12 等）或组合键（`⌘/⇧/⌥/⌃` 与主键组合，如 `⌘ + Enter`、`⌃ + ⇧ + Space`）
- 按键录入交互：点击"提交按键"字段进入捕捉模式 → 按下目标组合键 → 自动保存并退出；按 `Escape` 取消本次捕捉
- 设置关闭时完全不模拟按键，保持 v0.3 行为

**图片消息处理：**
- 飞书消息中含有图片时（单独图片或图文混合），Go sidecar 调用飞书 API 下载图片原始字节
- Rust 侧将图片写入 macOS **剪贴板**（`NSPasteboard`，`NSPasteboardTypeTIFF` / `NSPasteboardTypePNG`）
- 写入剪贴板后模拟 `Cmd+V` 触发粘贴，将图片粘入当前前台应用
- 若消息同时包含文本和图片：先注入文本，再粘贴图片（顺序保持消息原始顺序）
- 若目标输入框不支持图片粘贴（如纯文本终端），粘贴失败时写入日志，不影响文本注入

**不再做焦点校验**（v0.4.4+）：之前曾尝试通过 AX 查询焦点元素并拉 AXRole 白名单做"可输入性"判断，在 Electron / webview 类应用上频繁误判。改用剪贴板 + Cmd+V 策略后，彻底不再查焦点——**能否真的粘到目标由用户自己负责**。唯一兜底：通过 PID 对比判断当前前台是不是 TypeBridge 自己（防止字符打回自己的 webview），是就拒绝；否则粘贴命令照发，粘不到只是没反应，不再报"无焦点输入框"。

### 2.4 应用生命周期 / Dock + 托盘交互

应用同时在 Dock 和菜单栏托盘上展示入口（v0.7.x 起）。两者都只是"快捷入口"——单击直接打开主窗口、抢焦点。**托盘不挂下拉菜单**（参考微信桌面端的体验，避免冗余的"打开主窗口"菜单项）。

| 触发 | 行为 |
|------|------|
| **窗口左上角红色关闭按钮** | 拦截 close 事件，**隐藏窗口**（保留前端 React state）。应用进程不退出，Dock 和托盘图标都继续可见 |
| **Dock 图标单击** | 通过 `RunEvent::Reopen` 检测 `has_visible_windows == false` 时调 `show_or_create_main_window`，让窗口重新可见并抢焦点。修复 v0.7.x 之前 Dock click 不响应的 bug |
| **托盘图标单击** | 等价 Dock click，调同一个 `show_or_create_main_window`。托盘**不再有下拉菜单** |
| **Cmd+Q** | macOS 标准退出快捷键，正常退出进程 |
| **Dock 图标右键 → 退出** | macOS 标准退出路径，正常退出进程 |

**为什么去掉托盘下拉菜单**：原版菜单只有"打开主窗口"和"退出应用"两项，前者已经被单击行为覆盖，后者通过 Cmd+Q / Dock 右键退出已经能完成；保留菜单只是徒增一次右键交互成本，不带来真实的功能。

**托盘图标设计**：橙红渐变圆角底 + 白色桥拱图案（与 app icon 的白底橙桥配色反转），22pt×22pt 视觉面积适配 macOS 菜单栏标准高度。

### 2.5 主窗口（左侧侧边栏 + 内容区）

应用仅保留一个主窗口（820×560 px）。左侧 ~150px 竖向侧边栏，**所有 tab 平铺在顶部**（v0.7.x 起系统日志回归常规 tab，不再单独占底部 slot；连接状态指示也不再放底部冗余展示——已由「连接 TypeBridge」tab 内部的子 tab 承担）：

| Tab | 内容区 | 备注 |
|-----|--------|------|
| **连接 TypeBridge** | 顶部一句话说明 + 横向子 tab 切换（WebChat / 飞书 / 钉钉 / 企微）+ 当前渠道的配置表单 | 原 v0.5 的"服务配置"板块 + 3 个独立 tab，v0.6+ 合并为单一 sidebar tab + 内容页顶部横向 tab |
| **输入设置** | 输入后自动提交 + 提交按键 | |
| **历史消息** | chip 筛选（全部 / 飞书 / 钉钉 / 企微）+ 消息卡片列表 | v0.5 前叫"消息历史"，v0.5 起统一 |
| **系统日志** | 按时间倒序列表（含 `[渠道]` 彩色前缀） | 不含消息 / 注入事件（那些归历史消息 tab） |
| **关于 TypeBridge** | logo + 名称 + 版本号 + 检查更新按钮 | v0.7.x 新增；详见 §2.11 |

**连接状态展示（v0.7+）**：**不再**在侧边栏底部单独列每渠道的连接状态行。连接状态直接体现在「连接 TypeBridge」tab 内部的横向子 tab 上 — 每个子 tab 在连接成功时左侧会有一个绿色小圆点（WebChat 在 bound 态、IM 三家在 status=connected 态）。理由：信息在两处同步维护容易失同步；既然子 tab 本身就有状态指示，侧边栏底部的副本就是重复。

#### Tab 1: 连接 TypeBridge

内容区自上而下三部分：

**① 一句话说明**（顶部贯穿横条）：「通过内置 WebChat 或 IM 应用机器人连接 TypeBridge 进行输入」。`Info` icon + 文字，灰底低对比度，不抢注意力

**② 横向子 tab**（飞书 | 钉钉 | 企微）：点击切换下方的渠道配置面板；active tab 用 2px accent 色下划线 + 文字加深。子 tab 状态用 Zustand 记忆，切走 tab 再回来保留当前选中渠道

**③ 渠道配置面板**（占据剩余高度，独立滚动）：
- 飞书：顶部「还没有自建应用？」引导 banner → App ID / App Secret 输入 → 启动长连接 + 测试连接 → scope 检查清单
- 钉钉：结构对齐，字段名改 Client ID / Secret
- 企微：v0.6 暂占位，P3 接入智能机器人长连接后落地

#### Tab 2: 历史消息

展示所有收到并处理过的消息，按进入时间**倒序**排列（最新在最上方）。

**消息卡片**包含：
- 收到时间（相对时间：刚刚 / 2 分钟前 / 昨天 15:30 / 2026-04-30）
- 消息正文（文本 + 图片缩略图）
- 状态 tag：`已入队` / `处理中` / `已发送` / `失败`
- **失败原因分两个独立层级展示**，用户一眼看出各自属于哪个环节：
  1. **本地输入失败**（橙色小字，紧贴消息正文）：消息写入这一步失败。常见原因：`辅助功能权限未授予`、`当前前台是 TypeBridge 自己`、`剪贴板写入失败`。跟飞书无关
  2. **飞书反馈被拒**（红色 banner，放在"本地输入失败"下方）：机器人给这条消息发表情 / thread 回复时被飞书拒绝（scope 权限不足、频控、emoji_type 非法等）。这个是**双向通信**的问题，跟消息能否输入到本地输入框**完全独立**——消息可能已成功输入，只是用户在飞书里看不到机器人的反馈
- 两类错误可以**同时出现**：本地输入先失败 → 机器人想在飞书里告诉用户"失败了" → 发反馈又被飞书拒。卡片会同时展示两条信息，让用户分别定位每个层级的问题
- 卡片右侧操作按钮：
  - **复制**：把消息内容复制到系统剪贴板。点击后按钮形态切换为 `已复制` + 对勾图标 1.5s 再恢复，作为成功反馈。文本消息复制文本；纯图片消息复制图片字节到剪贴板；图文混合时优先复制文本（留图于剪贴板会覆盖上一条 pending 数据）
  - **删除**：从历史中移除该消息（硬删除，伴随移除本地图片缓存）

> 设计取舍：v0.4 原本有"重发"按钮（重新入队触发注入）。但实测点击按钮**本身**会把焦点从用户原本的目标输入框夺走 → 消息到达时又变成无焦点，一键死循环。所以把"重发"替换为"复制"，用户自己再把光标摆到目标输入框后粘贴，反而更可控。

**顶部"清空"按钮（硬清空 + 取消队列）**：
- 点击后弹出**应用内确认弹窗**（不是浏览器原生 `window.confirm`，因为 Tauri WKWebView 默认不实现该 panel，调用直接返回 false）
- 确认后会：清空 `~/.typebridge/history.json`、清空 UI 卡片，并把队列中尚未注入的对应条目标记为取消（已注入或正在注入的不受影响）
- 重启后被清空的消息**不会**再出现（底层数据已删除）
- 想精确删除某一条 → 用每条卡片上的「删除」按钮（同样会取消队列中该 id）

**空态**：无消息时显示提示文字 + 插画。

**持久化**：
- 消息历史写入 `~/.typebridge/history.json`（应用关闭后仍保留）
- 最多保留 **500 条**，超出自动滑动淘汰最早的（`FIFO`）
- 图片内容不内嵌，存到 `~/.typebridge/images/<message_id>.<ext>`，历史记录只存相对路径

#### Tab 5: 系统日志

展示应用连通性 / 错误 / 权限 / 启动事件等运维日志（**不含用户消息内容**，那些归"历史消息"tab）。保留"在访达中显示"按钮跳转到日志文件目录。

**顶部"清空"按钮（软清空）**：
- 仅清理当前 UI 中可见的日志条目；`~/Library/Logs/TypeBridge/*.log` 文件 **不动**
- 清空后新产生的事件仍正常展示；文件日志继续按天滚动写入
- 想看被清空的历史日志 → 用右侧"在访达中显示"跳转到日志文件目录查看

#### Tab 6: 输入设置

控制消息到达本地后的输入行为，v0.5 从「连接」tab 拆出，独立于飞书凭据配置。

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| **输入后自动提交** | 开启 | 开启后，消息写入焦点输入框后模拟按下"提交按键"（默认 Enter），完成一键发送；关闭时仅注入文本不触发提交 |
| **提交按键** | `Enter` | "输入后自动提交"使用的按键。默认 Enter；支持自定义为任意按键或组合键（如 `⌘ + Enter`、`⌃ + Enter` 等）。仅当"输入后自动提交"开启时生效 |

点击"提交按键"字段进入捕捉模式 → 按下目标组合键 → 自动保存并退出；按 `Escape` 取消本次捕捉。设置变更立即持久化到 `~/.typebridge/config.json`，无需重启 sidecar。

**为什么拆成独立 tab**：输入行为与"连接飞书 Bot"是两个独立关注点——用户第一次配置凭据时不需要关心输入设置，日常使用时调整输入设置也不应被"连接/测试连接"干扰。拆 tab 后「连接飞书 Bot」tab 纯粹承担"凭据 + 权限校验"职责，语义更清晰。

### 2.6 飞书消息双向回复

机器人在接收与处理消息的各阶段，向用户给出可视反馈：

| 阶段 | 回复形式 | 具体内容 |
|------|---------|---------|
| 收到消息（入队） | 表情反应 | 给原消息加 `EYES`（👀，"已看到"） |
| 处理成功（已注入） | 表情反应 | 将原有 `EYES` 替换为 `DONE`（✅，或飞书等价 emoji_type，具体取值见技术方案） |
| 处理失败 | 表情反应 + 文字 | 将原有 `EYES` 替换为 `SAD`（❌），并在原消息 thread 下回复一条文字消息说明失败原因（如：`无聚焦输入框`、`accessibility 权限未授予`、`用户取消`） |
| 处理中（长任务或等待确认） | 表情反应 | 保持 `EYES`，直到进入终态 |

**设计取舍：**
- 成功场景不额外发文字，避免污染群聊；失败场景必须发文字，方便用户明白为什么
- 表情替换优先"先加新 / 再删旧"顺序，避免短暂"无任何表情"视觉断层
- 表情反应失败（权限/网络问题）不影响主流程注入，仅记录到系统日志

### 2.7 消息队列机制

用户高频发消息时必须**严格串行**处理，避免多条消息同时注入打架：

- 所有收到的消息进入**单一 FIFO 队列**
- 同一时刻只有一条消息处于 `处理中`，其余 `已入队` 排队等待
- 处理完成（成功或失败）后立即出队并取下一条
- UI 中每条消息的状态 tag 实时反映当前所处阶段

**状态机：**

```
接收消息 → [已入队] ──worker 取出──► [处理中] ──┬─► [已发送]
                                              │
                                              └─► [失败]
```

**失败重试**：不自动重试。失败后消息保持 `失败` 状态，用户可在消息历史 tab 中点"重发"重新入队。

**并发硬约束**：严格单 worker 串行；注入期间新消息只入队不注入。

### 2.8 日志系统

**日志窗口内容（实时滚动）：**
- 飞书连通性日志（连接成功、断线、重连尝试）
- 收到的消息内容（发送人、时间、消息文本，敏感内容可选脱敏）
- 注入状态（目标应用名、注入成功/失败原因）
- 权限状态变化

**日志文件：**
- 写入路径：`~/Library/Logs/TypeBridge/typebridge-YYYY-MM-DD.log`
- 按天滚动，保留最近 30 天
- 日志窗口底部提供「在访达中显示」按钮，点击后用 `NSWorkspace.selectFile` 在 Finder 中高亮该文件
- **多渠道日志前缀**：每条日志带 `[飞书]` / `[钉钉]` / `[企微]` 渠道前缀，沉到文件 + UI 系统日志 tab 都能区分来源；与渠道无关的全局事件（启动、辅助功能权限等）不带前缀

### 2.9 多渠道支持（钉钉 / 企微）

> §2.1-§2.8 是以飞书为参考实现的细节描述；本节是钉钉 / 企微的接入需求 + 多渠道特有的产品行为约定。技术架构详见 [TECH_DESIGN §二十六](./TECH_DESIGN.md)。

#### 2.9.1 总体目标

**用户可以同时启用飞书 / 钉钉 / 企微三个渠道，每家一个机器人，三家收到的消息进入同一个 FIFO 队列，依次粘到当前焦点输入框**。各渠道彼此独立——一家断线不影响其他俩；用户也可以只配一家或两家，未配置的渠道不启 sidecar、不计入连接状态。

#### 2.9.2 各渠道接入要点

| 渠道 | 长连接技术 | 凭据 | 平台后台配置要点 |
|------|----------|------|--------------|
| **飞书** | `larkws` WebSocket | `App ID` (`cli_` 前缀) + `App Secret` | 自建应用 + 「事件配置」开启 `im.message.receive_v1` + 长连接验证 |
| **钉钉** | Stream Mode (反向 WSS) | `Client ID` (= AppKey) + `Client Secret` (= AppSecret) | 企业内部应用 + 添加"机器人"能力 + 消息接收模式选 Stream |
| **企微** | AI Bot 长连接 (`wss://openws.work.weixin.qq.com`) | `Bot ID` + `Secret`（长连接专用，与回调 token 不互通） | 智能机器人 + API 模式切"长连接"（与回调互斥） |

**全部零公网 IP / 零回调 URL 依赖**——这是选用三家"长连接"模式的核心动因：桌面 macOS 应用不可能稳定暴露公网 URL，传统 webhook callback 路线在用户侧落地复杂度过高。

#### 2.9.3 渠道能力差异（用户可见差异）

| 用户可见行为 | 飞书 | 钉钉 | 企微 |
|------------|------|------|------|
| 收到消息时机器人侧给反馈 | EYES 表情（👀） | 不单独回执（注入 <1s） | 流式消息 `🟡 处理中...`（stream.id 原地更新） |
| 处理成功反馈 | DONE 表情（✅） | 文字回执 `✅ 已输入` | 同一条流式消息原地更新为 `✅ 已输入` |
| 处理失败反馈 | CRY 表情 + thread 文字回复 | 文字回执 `❌ 输入失败：<原因>` | 同一条流式消息原地更新为 `❌ 输入失败：<原因>` |
| 接收图片消息 | ✓ | ✓ | ✓ |
| 群 @ 机器人 / 单聊 | 都支持 | 都支持 | 都支持（单聊 + 群聊都走长连接 callback） |

**反馈机制差异说明**：三个渠道对应三种形式的反馈实现，抽象到 Rust 侧的 `ChannelCapability` 能力矩阵。

- **飞书** `reactions` + `failure_text_reply`：emoji 直接贴原消息，不产生新消息；只有失败时额外发一条 thread reply 说明原因。
- **钉钉** `success_text_reply` + `failure_text_reply`：协议级**不支持 bot reaction 也不支持流式更新**，退化为 `sessionWebhook` 发一条文字回执（成功或失败）。"接收中间态"（<1s）不单独回执以免刷屏。
- **企微** `streaming_reply`：利用 `aibot_respond_msg` 的 `stream.id + finish` 机制，同一条 bot 消息原地从 `🟡 处理中` 更新为 `✅ 已输入` / `❌ 输入失败`，体验上等价于飞书 reaction（一条消息状态原地变）。

precedence（queue.rs）：`streaming_reply > reactions > success_text_reply / failure_text_reply`，互斥，每个渠道只走一条反馈路径。

#### 2.9.4 主窗口侧边栏「服务配置」板块

侧边栏「服务配置」板块下三个 tab 都是**真实可用的**配置 + 测试 + 启动 tab（v0.5 之前钉钉 / 企微为占位）：

- **连接飞书 Bot**（参考实现）
- **连接钉钉 Bot**：包含 Client ID / Secret 输入 / 启动 / 测试 + Stream Mode 配置引导 banner
- **连接企微 Bot**：包含 Bot ID / Secret 输入 / 启动 / 测试 + API 模式切换引导 banner

每个 tab 内的 UI 结构高度对齐：顶部引导 banner（去对应平台后台）→ 凭据输入 → 启动/测试按钮 → 连接状态 → 测试连接清单。

**顶部 banner 按连接状态切换**（v0.7+）：

| 状态 | 文案骨架 | 作用 |
|------|---------|------|
| **未连接 / 启动中** | "还没有 XX 应用？先到 XX 开发者后台……"（保留原有配置引导） | 让首次用户知道下一步要去哪里拿凭据 |
| **已连接** | 「XX 机器人已就绪。打开 XX App 找到你配置的机器人，发消息即自动注入到桌面当前焦点输入框。」+ 小贴士：「TypeBridge 不接收语音消息。想用语音输入，请使用 XX 的「语音转文字」功能或手机输入法的「语音输入」功能。」 | 告诉用户接下来可以做什么；提前说明语音不能直接发，避免用户试错 |

三家的小贴士文案**完全同构**，只替换 XX 名（飞书 / 钉钉 / 企业微信）。设计理由：三家的语音功能交互高度一致（都是对话框麦克风按住说话），特意用相同句式能降低用户在不同渠道间切换时的认知负担；"或手机输入法的语音输入功能"是兜底，保证即使用户的 IM 版本没有开启云转文字也能走通流程。

#### 2.9.5 历史消息：渠道筛选 + 来源 tag

历史消息 tab 顶部 chip 筛选器：

```
[ 全部 12 ] [ 飞书 8 ] [ 钉钉 3 ] [ 企微 1 ]
```

- 默认选中"全部"
- 点击单个渠道只展示该渠道消息
- 计数实时跟随历史变化
- **未配置过的渠道不显示对应 chip**——避免空 chip 干扰

每张消息卡片右上角 **渠道 tag**（小图标 + 文字 + 配色）：
- 飞书：`Plug` icon + "飞书" + 主题色
- 钉钉：`Plug` icon + "钉钉" + 蓝色调
- 企微：`Plug` icon + "企微" + 绿色调

#### 2.9.6 连接状态展示（v0.7+）

连接状态**只在「连接 TypeBridge」tab 内部的横向子 tab 上展示** — 每个子 tab 连接成功时 label 左侧有一个绿色小圆点：

- WebChat：`phase === "bound"`（有 ≥1 台手机绑定）
- 飞书 / 钉钉 / 企微：`typebridge://status` 事件的 `connected === true`

WebChat 的 channelConnected 由 MainWindow 订阅 `typebridge://webchat-session-update` 事件维护；IM 三家由 `typebridge://status` 维护。两条路径最终都写进 Zustand 的 `channelConnected[ch]`，子 tab 统一读取。

侧边栏底部**不再**单独列每渠道的连接状态行，原 v0.6 的底部状态列表已删除（信息在两处同步维护容易失同步；子 tab 本身已有状态指示就够了）。

#### 2.9.7 系统日志渠道前缀

每条系统日志按 `[渠道] 内容` 格式：

```

14:23:50 [钉钉] 收到消息 msg_xyz123
14:24:10 [企微] 重连尝试 1/∞，2s 后重试

```

UI 系统日志 tab 不引入 chip 筛选（保持简洁），用户靠肉眼扫前缀分类即可；后续如有强需求再加。

#### 2.9.8 边界情况清单

| 场景 | 行为 |
|------|------|
| 用户只配置 1 个渠道 | 仅启对应 sidecar，其他渠道 tab 仍可见但无需启动；侧边栏底部状态只显示已配置那个 |
| 三个渠道同时收到消息 | 全部进入同一 FIFO 队列；按到达顺序串行处理；每条消息 `channel` 字段记录来源 |
| 单渠道断连 | 该渠道 sidecar 自动指数退避重连（其他渠道不受影响）；历史消息上的反馈卡片可能因断连未更新成功 |
| 凭据错误 | 该渠道 sidecar 启动失败，连接状态展示原因；不影响其他渠道 |
| 用户清空某渠道凭据 | 停止对应 sidecar，状态变"未配置"；历史里的旧消息保留（仍按 channel tag 区分展示）|
| 历史消息 ID 跨渠道冲突 | 不会冲突——`HistoryMessage.id` 用复合键 `{channel}:{source_id}`（详见 TECH_DESIGN）|

### 2.10 WebChat 渠道（本地局域网网页机器人）

> 与 §2.9 三家 IM 不同，WebChat 是 **TypeBridge 自带** 的渠道：用户不用任何企业 IM 账号，桌面 App 在本机启动一个 HTTP + WebSocket 服务，同 WiFi 下的手机扫码即可发消息。**完全不依赖任何云端服务**。技术细节见 [TECH_DESIGN §三十五](./TECH_DESIGN.md)。

#### 2.10.1 总体目标

为没有企业 IM 账号 / 不想暴露 IM 凭据 / 仅想快速试用的个人用户提供"开箱即用的输入桥"。整体形态：

- 桌面 App `连接 TypeBridge → WebChat` tab 点"启动会话"，App 在本机起一个 HTTP server（固定 8723，占用则递增 fallback）
- UI 展示 **二维码 + 6 位 OTP + 本机 WiFi 名称**
- 同 WiFi 下手机扫码 → 进入聊天页（SPA 静态资源从本机 server 提供）→ 输入 OTP 完成握手
- 发文本 / 图片 → 通过 Socket.IO 实时送到桌面 → 注入到当前聚焦输入框
- 桌面端注入完成后回手机一条 ack（"已注入"或"失败原因"）

**核心前提**：手机和电脑必须在**同一个 WiFi / 局域网**。如果用户手机不方便切 WiFi，UI 会引导改用飞书/钉钉/企微机器人渠道。

WebChat 与飞书 / 钉钉 / 企微 **完全平等**：可以同时启用，消息进入同一个 FIFO 队列，历史与日志都带 `webchat` 渠道标签。

#### 2.10.2 用户流程（happy path）

1. 桌面打开 TypeBridge → `连接 TypeBridge` → `WebChat` tab
2. idle 态：一条简短的操作提示 + "启动会话"按钮（**不展示 WiFi 提醒**，见 §2.10.6）
3. 点"**启动会话**" → App 启动本机 HTTP + Socket.IO server → 显示 QR / OTP / 60 秒倒计时进度条
4. QR 码编码 `http://<本机 LAN IP>:<port>/?s=<sessionId>`
5. 用户掏手机扫码 → 手机浏览器加载 SPA → 握手页面 → 输入 6 位 OTP → 点确认
6. Socket.IO 本地校验 OTP → 签发 userToken 给手机 → 切换到聊天页
7. 用户在手机上发消息 → Socket.IO 送到桌面 → 进入 FIFO 队列 → 注入到当前聚焦输入框 → Socket.IO ack 回手机
8. 手机消息流上每条消息显示 `已注入` / `失败：<原因>` 状态徽标

**OTP 自动轮换**：60 秒倒计时归零时，桌面**自动生成新的 6 位 OTP**（无需用户点任何按钮），倒计时进度条重置。**已绑定的手机不受影响**（server 按 userToken 校验后续消息，不关心 OTP 是否过期）。只有未握手的新手机需要看新 OTP。轮换是纯更新 OTP + 重置倒计时，**不重启 server、不中断已有连接**。60 秒足够一个熟练用户完成"看屏 → 拿手机 → 扫码 → 输 OTP"全流程，同时把未使用 OTP 的有效窗口控制得足够短。

#### 2.10.3 安全模型

- **OTP 与 sessionId 分离**：QR 码里只有 sessionId，OTP 由桌面屏幕单独显示。物理上必须同时看到电脑屏幕和手机才能完成握手
- **服务仅本地可达**：server 绑在局域网 IP，同 WiFi 外的用户无法访问
- **OTP 60 秒自动轮换**：OTP 倒计时归零时桌面无感生成新 OTP + 重置倒计时；已绑定手机继续用 userToken 发消息，不受影响；只影响尚未握手的新手机
- **OTP 5 次错误锁定**：手机端连错 5 次自动作废 OTP，桌面 UI 切换到"验证码已锁定"态，用户手动点"重置 OTP"按钮生成新 OTP 解锁（**不会踢掉已绑定的手机**）
- **多手机共享**（新）：同一 sessionId 允许同 WiFi 内多台手机分别输 OTP 绑定，各自签发独立 userToken；消息都进同一 FIFO 队列
- **桌面 App 关闭即会话终止**：server 进程退出，所有 Socket.IO 连接断开，手机端会显示"桌面已断开"并退出聊天页
- **OTP 不明文存储**：桌面内存里保留 sha256 + 明文（UI 展示用），持久化层不写
- **无云端依赖**：没有任何数据经过线上；Netlify 旧部署已彻底下线

#### 2.10.4 消息形态

| 类型 | 手机端输入方式 | 注入到桌面 | 备注 |
|------|--------------|----------|------|
| 文本 | 键盘 | 文本注入到聚焦输入框 | 与飞书 / 钉钉 / 企微完全一致 |
| 图片 | 拍照 / 相册（仅 JPEG/PNG） | 图片粘贴到聚焦输入框 | 客户端压缩到 ≤2MB；iOS HEIC 自动转 JPEG（系统级） |
| 语音 | 直接用手机输入法的麦克风按钮（输入法自带的语音转文字） | 识别后的文字按文本路径注入 | WebChat **不自研语音识别**，也**不再在 WebChat 内部放语音入口按钮 / 引导弹层**——用户在手机键盘里点麦克风即可，无需 WebChat 中转。好处：零包体膨胀 / 完全兼容 IM 浏览器 / 无需 HTTPS / 不引入"伪入口"造成误解 |

**单向 + ack**：消息只从手机 → 桌面；桌面只回"注入成功 / 失败"作为 ack 状态徽标，不发反向消息。

##### 2.10.4.1 控制键快捷面板（手机端）

WebChat 移动端在输入框上方提供一排"控制键"快捷按钮，让用户通过手机直接控制桌面键盘行为（不输入文本，只触发按键事件）。

**面板位置与开合**：
- 位置：输入栏（ComposerBar）正上方
- 默认状态：**收起**。"展开/收起"切换按钮**内嵌在输入栏右侧、原语音按钮的位置**（语音按钮已下线，参见 §2.10.4），不再以悬浮 tab / 箭头形式出现在输入栏外部
- 切换按钮使用 lucide **`Keyboard`** 图标（语义直接对应"控制键面板"）。激活态（面板已展开）按钮整体填充 accent 色 + 白色图标；未激活态使用普通底色 + 中性图标色。**不再使用 ChevronUp/Down 箭头**
- 用户点击切换按钮 → 展开/收起按键面板；展开/收起状态记入 `localStorage`，下次进入会话沿用

**按键集合与排列顺序（v1）**：

按钮从左到右严格按照「**上 → 下 → 左 → 右 → 空格 → 回车 → 删除**」排列。删除键放在最右侧（手指最易触达且需要明确意图），方向键放在最左侧形成视觉聚类。

| 按键 | 行为 | 备注 |
|------|------|------|
| 上 / 下 / 左 / 右 | 方向键 | 用于光标移动 / 列表选择 |
| 空格 Space | 在桌面焦点输入框按下 Space | 输入空格 |
| 回车 Enter | 在桌面焦点输入框按下 Return | 用于发送 / 换行 / 确认 |
| 删除 Backspace | 在桌面焦点输入框按下 Backspace | 删除前一个字符 |

**与文本消息的执行顺序**：
- 控制键事件**与文本消息共用同一 FIFO 队列**。如果用户先发了一段文本（仍在粘贴中），紧接着点回车，回车会在文本注入完成后才触发。这避免回车插在粘贴中间导致提前提交。

**视觉规范**：
- 按钮使用 lucide 图标（CornerDownLeft / Delete / Space / ArrowUp/Down/Left/Right），不使用 emoji 或 unicode 装饰符
- 单行水平排列，按钮间均匀间距；按钮 hit area ≥ 36×36 pt，符合手指点按
- 按下时有触觉反馈视觉（背景色短暂变深），与现有 ComposerBar 视觉语言一致
- 与桌面"输入设置"中的 SubmitKey 配置**无关**：那是注入完成后的自动提交按键，这是用户主动触发的独立按键

**安全**：
- server 侧维护按键白名单（仅 Enter / Backspace / Space / Arrow 系列等无副作用的导航控制键），任何不在白名单的 key code 一律拒绝。**不允许** Cmd / Ctrl / Option / 字母数字等可能引发任意操作的键，避免成为远程任意按键执行通道。

#### 2.10.5 入口阻断

WebChat 网页仅对 **PC 浏览器**做硬性引导，**IM 内置浏览器（微信 / 钉钉 / 飞书 / QQ）全部放行**：

- **PC 浏览器**（Mac / Windows / Linux）：显示"请用手机访问"全屏引导页，附说明：WebChat 是手机端输入桥，桌面用 PC 自己给自己发没意义
- **IM 内置浏览器**：无拦截。由于新架构不再需要 `getUserMedia` / Web Speech API，IM 内置 WebView 的 API 限制不再影响功能，用户可以直接用

UA 检测在 SPA 内做客户端判断（HTTP 模式无 server component）：PC 分支渲染拦截页，其他情况进正常会话流程。

#### 2.10.6 桌面端配置 tab

`连接 TypeBridge → WebChat` tab 的 UI 结构（与现有飞书 / 钉钉 / 企微 tab 风格保持一致）：

**① 启动前（idle 状态）** — 只有一条纯提示 + 一个启动按钮，不显示 WiFi 提醒（因为还没到扫码环节，提前提 WiFi 是噪音）：

```
┌─────────────────────────────────────────┐
│ ⓘ 点「启动会话」后，App 会在本机起一个  │
│   局域网 HTTP 服务，生成二维码 + 6 位   │
```

提示用扁平 tip 样式（图标 + 正文 + 浅底边框），**不加"会话状态"之类的 section 标题** — 在启动前把这里包装成"某个字段"会显得过度设计。

**② 启动后（session live — pending / bound 共用同一视图）** — QR 和 OTP 在整个会话周期都保持可见，方便**追加**的新手机随时扫码加入（多设备场景，§2.10.3）。仅头部的"WiFi 提醒 vs. 已连接 N 台设备"横条根据是否已有设备绑定切换：

**② -a 尚未有设备绑定时（pending 状态）**：

```
┌─────────────────────────────────────────────┐
│ ⓘ 扫码之前确保手机与本机连接到同一 WiFi     │
│                                             │
│ ┌─────────────┬─────────────────────────┐   │
│ │             │ ● OTP 验证码       [↻] │   │
│ │             │ ┌─────────────────────┐ │   │
│ │  [QR 160]   │ │    5 2 7 0 9 3      │ │   │
│ │             │ ├─────────────────────┤ │   │
│ │             │ │█████████████────────│ │   │← 底部进度条装饰
│ │             │ └─────────────────────┘ │   │
│ │ 用手机扫码  │ 剩余 00:45               │   │
│ │ http://...  │                          │   │
│ └─────────────┴─────────────────────────┘   │
│                                             │
│ 手机扫码后输入此 6 位 OTP 完成绑定。         │
│ 验证码每 60 秒自动轮换，已绑定的手机不受影响。│
│                                             │
│ [⏸ 停止]                                     │
└─────────────────────────────────────────────┘
```

**② -b 已有 ≥1 台设备绑定时（bound 状态）** — QR + OTP + 进度条**完全照旧**（其他手机仍可继续扫码加入），仅顶部横条换成"已连接"提示：

```
┌─────────────────────────────────────────────┐
│ ✓ 已连接 2 台设备。QR / OTP 继续有效，       │
│   其他手机扫同一码即可加入                   │
│                                             │
│ ┌─────────────┬─────────────────────────┐   │
│ │  [QR 160]   │ ● OTP 验证码       [↻] │   │
│ │             │ ┌─────────────────────┐ │   │
│ │             │ │    5 2 7 0 9 3      │ │   │
│ │             │ ├─────────────────────┤ │   │
│ │             │ │█████████████────────│ │   │
│ │             │ └─────────────────────┘ │   │
│ │ 用手机扫码  │ 剩余 00:45               │   │
│ │ http://...  │                          │   │
│ └─────────────┴─────────────────────────┘   │
│                                             │
│ [⏸ 停止会话]                                 │
└─────────────────────────────────────────────┘
```

**设计要点（v0.7+）**：

- **横排 QR + OTP**：QR 从原 208px 缩到 160px（手机摄像头 100px+ 即可扫），腾出右侧放 OTP 列，信息密度提升约 2 倍
- **OTP 单容器展示**：不再是 6 个独立方框，一个圆角 pill 里用 `font-mono` + 字符间距拉开；视觉更像"一串号码"而非"输入位格"
- **进度条融入容器**：底部 3px 装饰条，宽度从 100% 平滑缩到 0%，不再单独占一行
- **手动刷新按钮**：OTP label 行右侧一枚 `RotateCw` 图标按钮，点击立刻调 `rotate_webchat_otp`（原子替换 OTP hash + 重置倒计时）。用于"看这个 OTP 不爽/怀疑泄露/想马上换"的手动兜底，日常靠自动轮换即可

**③ 验证码锁定（expired 状态，OTP 5 次错）**：

```
┌─────────────────────────────────────────┐
│ ⚠ 验证码已锁定                          │
│ 连续 5 次 OTP 错误，当前验证码已锁定。  │
│ 点下方按钮生成新 OTP 解锁（已绑定设备   │
│ 不会受影响）。                          │
│                                         │
│ [⟳ 重置 OTP]                             │
└─────────────────────────────────────────┘
```

设计原则：
- **不再有"重启会话"按钮**。"重启"这个动作实际上会踢掉所有已绑定的手机，语义过于重。用户真实需求只有两个：换个 OTP（自动或手动）/ 彻底停掉 server，对应的动作都已拆分清楚
- OTP 轮换（倒计时归零）是**桌面无感触发**的，用户不需要点任何按钮
- "重置 OTP" 仅在锁定态出现，作为手动兜底

握手成功瞬间：桌面 App 自动隐藏菜单栏窗口 + 系统通知"WebChat 已连接，请把焦点切到目标输入框"，避免 `is_frontmost_self` 误伤导致首条消息注入失败。

#### 2.10.7 边界情况

| 场景 | 行为 |
|------|------|
| 用户在 WebChat tab 停留时手机发消息 | 桌面端窗口自动隐藏，焦点回到上一前台 App；若仍在 TypeBridge，显示通知"请把焦点切到目标输入框" |
| 端口 8723 被占用 | 递增尝试 8724/8725/.../8732，全占则 UI 报错"端口 8723-8732 全被占用，请关闭其他应用重试" |
| 手机切换网络（WiFi 断开 / 切 4G） | Socket.IO 客户端自动重连；若 30s 内未恢复，聊天页显示"连接中断，请检查 WiFi" |
| 桌面 App 退出 | server 进程停止，Socket.IO 连接断开；手机端显示"桌面已关闭，请在桌面重新启动会话" |
| OTP 60 秒倒计时归零 | 桌面自动轮换 OTP + 重置进度条；**已绑定设备不受影响**；正在握手的新手机需重新输入新 OTP |
| OTP 5 次错误 | 本机校验锁定当前 OTP；手机端显示"验证码错误次数过多，请桌面端重置验证码"；桌面 UI 切到"验证码已锁定"态，用户点"重置 OTP"按钮恢复 |
| 同 sessionId 多台手机扫码 | 各自输正确 OTP 后都能连上，消息 FIFO 串行注入（新语义） |
| 用户在 PC 浏览器输入完整 URL | 客户端 UA 检测渲染"请用手机访问"页面 |
| 用户用微信 / 钉钉浏览器扫码 | 正常进入握手页，不再拦截（新语义） |
| 手机想用语音 | WebChat 不再提供语音入口（按钮和引导弹层已移除）。用户直接点手机键盘上的麦克风按钮用输入法语音输入即可——识别后的文字按文本路径正常注入桌面 |
| 手机离线 / 切后台 | Socket.IO 重连成功后幂等重发；未送达的消息 UI 显示"重发"按钮 |

- 历史消息 chip 筛选器：`[ 全部 N ] [ WebChat ] [ 飞书 ] [ 钉钉 ] [ 企微 ]`
- 历史卡片右上角渠道 tag：`Smartphone` icon + "WebChat" + TypeBridge 品牌橙

- 侧边栏底部状态：`WebChat ● 已连接 (2 台设备)`

#### 2.10.9 分发 / 打包

- **前端 SPA** (`webchat-local/dist/`) 作为 Tauri resource 打包进 `.app`
- **本地 server** 由 Tauri Rust 进程内嵌 axum + socketioxide 实现，无外部二进制
- **无语音模型 / 无 WASM 引擎**：.app 体积不受 WebChat 影响（小于 1MB 前端 SPA + ~200KB Rust 依赖增量）
- 用户完全离线也能使用：下载 `.app` → 启动 → 扫码 → 发消息。**不需要任何网络请求**
- **DMG 安装界面必须清晰且有品牌感**：官网下载安装包后，Finder 挂载窗口展示定制背景、明确的“拖拽到 Applications”安装指引、左右两个视觉落点和品牌说明，避免默认空白窗口或元素错位造成的廉价感。背景图不重复绘制 App 名称 / Applications 名称，名称交给 Finder 图标标签显示；背景预留区域必须与真实图标落点对齐，并使用 Retina 2x 资源保证清晰度。底部必须提示当前应用未经过 Apple 公证，以及首次启动前到“系统设置 > 隐私与安全性”点击“仍要打开”的路径

#### 2.10.10 隐私声明

WebChat 渠道**完全不走任何线上服务**：

- **消息不出本机**：所有文本 / 图片通过局域网 Socket.IO 传到桌面，不经过任何中继
- **无 ASR / 无模型**：语音识别由用户手机自带输入法完成，TypeBridge 本身不处理音频
- **OTP 不持久化**：桌面内存里保留 OTP 明文用于 UI 展示，进程结束即消失；持久化层只存 sha256
- **无追踪 / 无分析 SDK**：前端不接入任何统计 / 分析 / 广告 SDK
- **开源可审计**：server 端完整实现在 `src-tauri/src/webchat_server.rs`，用户可自行审阅源码

页面底部固定挂一条简短声明："WebChat 服务运行在你的电脑本地，数据不经过任何线上服务，请安心使用"。

#### 2.11.1 页面布局

#### 2.11.2 检查更新流程

```
点检查更新
  ├─ dev 构建 → toast/inline："当前版本已是最新版"（不打网络）
  └─ release 构建 → 调 Rust command → fetch 官网 API
       ├─ API 返回错误 / 网络异常 → toast：失败原因（保留按钮可重试）
       ├─ latest <= current → toast："当前已是最新版本 vX.Y.Z"
       └─ latest > current → 弹「确认更新」对话框
              "发现新版本 vX.Y.Z（当前 vY.Y.Y）。
               点击确认后：应用将退出 → 自动下载新版 .dmg → 在 Finder 中打开。
               请按 macOS 标准方式将新版拖入应用程序文件夹覆盖旧版。"
              [取消] [确认更新]
                       ↓
                 触发 apply_update：
                   ① fetch download_url → 写入 ~/Downloads/{filename}.dmg
                   ② 调系统 `open <dmg>` 挂载并打开 Finder 卷
                   ③ app.exit(0)
```

**为什么是半自动而非全自动重启**：完整 auto-update（download + 替换 .app + relaunch）需要 Tauri updater 插件 + 代码签名链路，签名要 Apple Developer 证书 / 自签 ed25519 密钥对 + CI 集成。当前 v0.7.x 阶段优先把链路打通，签名基建放后续版本（见 §2.11.5）。

- **重启不全自动**：用户仍需手动拖入 .dmg 完成安装。完整体验需要走 `tauri-plugin-updater` + 自签 / Apple 公证。

- **dev 构建的 `dev:latest` 是 placeholder**：不参与版本比较，永远显示"已是最新版"，避免开发期被自动检查打扰。

## 五A、各渠道接入清单

每个渠道接入时需要在对应平台后台完成的最小动作 + 所需权限。三家粒度差异较大，分开列出。

### 五A.1 飞书

跑通完整消息链路（接收 → 入队 → 粘贴 → 表情反应 → 失败时 thread 回复）共调用 **4 组飞书 API**，对应需要在开发者后台勾选的 scope 如下：

| 环节 | 触发 API / 事件 | 最小粒度 scope（推荐，2024 后新命名） | 兼容"大包" scope |
|------|----------------|----------------------------------|----------------|
| **1. 接收消息事件** | event `im.message.receive_v1`（WebSocket） | 按场景选：`im:message.p2p_msg`（单聊机器人）+ `im:message.group_at_msg`（群 @ 机器人） | `im:message` |
| **2. 下载图片资源** | `Im.MessageResource.Get` | `im:message:readonly` | `im:message` |
| **3. 发表情反应** | `Im.MessageReaction.Create` | `im:message.reactions:write_only` | `im:message` |
| **4. 回复消息**（失败 thread 回复） | `Im.Message.Reply` | `im:message:send_as_bot` | `im:message` |

> "大包" `im:message` 单独勾一个就能覆盖 2/3/4 全部需求；遵循最小权限原则时用左列三个细粒度 scope。

**重要说明：**
1. 接收消息事件（#1）不能通过 API probe 自动校验，必须由用户**在开发者后台「事件配置」页面**完成两步：①「订阅方式」选"使用长连接接收事件"并完成验证；②「添加事件」搜索 `im.message.receive_v1` 并勾选提交。提交事件时飞书会自动把对应 scope 加入应用，无需在「权限管理」里单独勾。TypeBridge 只能在 UI 中给出提示和深链，不能替用户操作这一步
2. 应用不需要任何 `im:chat*` / `contact:*` / `application:*` 权限。`im:chat:readonly` 过去仅作为 selftest 的"探活 API"绑架需要，v0.5 移除此依赖——selftest 改用消息链路真实需要的 API 做 probe
3. 所有 scope 勾选后，必须在开发者后台**发布一次新版本**才会生效（自建应用正式版/测试版皆可）
4. 凭据本身（App ID / App Secret）能换到 `tenant_access_token` 是所有调用的前提——probe 若返回 `code=99991663 invalid app_id` / `99991664 invalid app_secret`，清单整体判失败、直接提示去后台检查凭据

### 五A.2 钉钉

钉钉的"机器人收消息"能力**不需要勾任何 scope**——创建"企业内部应用"+ 添加"机器人"能力 + 选择 Stream Mode 后开箱即用。

需要在开发者平台 [open-dev.dingtalk.com](https://open-dev.dingtalk.com) 完成：

| 环节 | 操作 | 自动校验 |
|------|------|---------|
| **1. 创建应用** | "企业内部应用" → 复制 `Client ID` + `Client Secret` | UI selftest 中"凭据可用"项验证 |
| **2. 添加机器人能力** | 应用页 → 左侧"功能/机器人" → 添加 | 不可自动校验，纯静态引导 |
| **3. 选择消息接收模式** | 机器人页 → "消息接收模式" → 选 **Stream 模式** | 不可自动校验（无 introspection API），纯静态引导 |

**消息链路涉及的钉钉 API**：
- 接收消息：通过 dingtalk-stream-sdk-go `RegisterChatBotCallbackRouter` 自动订阅，无需 scope
- 下载图片：`/v1.0/robot/messageFiles/download`（用 `downloadCode`）—— 内置在 SDK，无需额外 scope
- 状态反馈：`SimpleReplyText` 通过 `data.SessionWebhook` 发 bot 文字回执 —— 内置在 SDK，无需额外 scope

**钉钉的状态反馈策略**：钉钉没有 bot reaction API，靠 `sessionWebhook` 发回执文字让用户知道消息被消费了。每条用户消息最多对应一条 bot 回执：注入成功 → `✅ 已输入`；注入失败 → `❌ 输入失败：<原因>`。"接收中间态"（<1s）不单独回执以免刷屏。飞书侧因为有 reaction 能力所以不发文字回执，只在原消息上贴表情。

**关键差异**：钉钉 Stream Mode **仅限组织内部使用**——其他企业的人无法触发该 bot；这与飞书自建应用语义一致。

### 五A.3 企微

企微的"智能机器人"能力**配置最简**——管理后台开启"长连接"模式 + 复制 `Bot ID` + `Secret` 即可。

需要在企微管理后台 [work.weixin.qq.com](https://work.weixin.qq.com) 完成：

| 环节 | 操作 | 自动校验 |
|------|------|---------|
| **1. 创建智能机器人** | "应用管理" → "智能机器人" → 创建 | UI 引导用户跳转 |
| **2. 切换 API 模式** | 智能机器人详情 → API 模式 → 选 **长连接**（与"回调模式"互斥）| 不可自动校验 |
| **3. 复制凭据** | 同页面复制 `Bot ID` + `Secret` | 通过实际 WSS 鉴权握手验证 |

**消息链路涉及的企微 API（全部走 WSS 帧，不是 HTTP）**：
- 接收消息：`aibot_msg_callback` 帧
- 下载媒体：每条媒体帧自带 `aeskey`，按 `AES-256-CBC + PKCS#7` 解密（不复用回调模式 EncodingAESKey）
- 状态反馈：`aibot_respond_msg` 帧（msgtype: `streaming` 实现"处理中→已完成"）

**关键限制**：
- **每个机器人同时只能一条活动 WSS 连接**——新连接踢旧连接。这意味着用户不能同时在多台 Mac 上运行 TypeBridge 接入同一个企微机器人
- 心跳每 30s 一次，断连后自动重连
- 切换为长连接模式后**回调 URL 失效**——这两种模式互斥，用户不能两个都用

## 八、CI/CD 发布流水线

### 8.1 目标

支持通过 GitHub Actions 在 macOS 云端环境中完成双架构（arm64 + x86_64）构建，自动生成 GitHub Release 并挂载 `.dmg` 产物，确保**任意版本可重复构建、重复发布**。

### 8.2 触发方式

提供两种互补的触发方式，各满足不同场景：

| 方式 | 触发条件 | 用途 | 可重复？ |
|------|---------|------|----------|
| **手动触发** | GitHub Actions UI 中点击 `workflow_dispatch`，输入版本号 | 日常开发迭代、测试构建、需要重建某历史版本 | ✅ 无限次；若同版本号 release 已存在则原地更新 |
| **Tag 自动触发** | 推送 `v*` 格式的 tag（如 `v0.2.0`） | 正式发布：打 tag 即自动构建 + 创建 Release | ✅ 删掉 tag 重新推送即可重建 |

两种方式共享同一条流水线，唯一的区别是版本号来源：
- 手动触发：从 `workflow_dispatch` 的 `version` 输入字段读取
- Tag 触发：从 `github.ref_name` 推解析（去掉 `v` 前缀）

### 8.3 产物

每次成功构建产出：

- `TypeBridge_<version>_aarch64.dmg` — Apple Silicon 安装包
- `TypeBridge_<version>_x64.dmg` — Intel 安装包
- 两个 `.dmg` 挂载在对应 GitHub Release 的 Assets 中

### 8.4 版本号覆写

构建前，CI 脚本自动用传入的版本号覆写以下位置：

- `src-tauri/tauri.conf.json` → `version`
- `src-tauri/Cargo.toml` → `package.version`

确保任意版本号同一份代码都能正确打出（不依赖开发者本地手动改版本号）。

## 八A、桌面端国际化（i18n，v0.8）

### 目标

桌面 App 全量支持简体中文（`zh`）/ 英文（`en`）双语，覆盖所有用户可见文案：SideBar 导航、所有 tab 内容、模态弹窗（辅助功能引导、首次启动语言选择）、通知、日志条目模板、错误提示。

### 用户路径

1. **首次启动**：未持久化语言时，主窗口正中央弹出语言选择卡片（中文 / English 两个等权按钮），不可绕过。用户选择后立即应用并写入持久化存储
2. **后续启动**：直接读取持久化语言渲染界面，不再弹出选择
3. **二次切换**：SideBar 左侧底部（在「关于 TypeBridge」入口下方）增加一个语言切换控件——以 `lucide Languages` 图标为触发点，点击展开「中文 / English」下拉菜单，当前语言带 `Check` 图标标记。切换后立即生效（无需重启）并持久化
4. 语言切换不重启 App、不重连 sidecar、不影响连接状态

### 翻译范围

| 区域 | 是否翻译 | 说明 |
|------|---------|------|
| SideBar 标签 + 底部入口 | 是 | 连接 TypeBridge / 输入设置 / 历史消息 / 系统日志 / 关于 |
| 各 tab 全部正文 | 是 | 含 ConnectionTab / WebChat / 飞书 / 钉钉 / 企微 / Input / History / Logs / About |
| 模态弹窗 | 是 | 辅助功能引导、首次语言选择、内联提示 |
| 通知（系统通知） | 是 | inject 成功/失败、错误提示 |
| 日志/历史中的「用户输入文本本身」 | 否 | 这是用户/对端发的内容，与界面语言无关 |
| 平台凭据字段名（AppID / AppSecret 等） | 否（保留英文术语） | 与第三方平台后台一致，避免歧义 |
| commit message / 内部日志文件 | 否 | 不在用户界面，按既有约定（中文为主）继续 |

### 持久化与默认值

- 语言首选项与其他用户设置一并持久化到 `~/Library/Application Support/com.parksben.typebridge/config.json`，新增字段 `language: "zh" | "en"`，默认空字符串代表「未选择」
- 已存在的存储升级时该字段缺省为空，触发首次启动语言选择流程（与全新安装行为一致）
- 默认建议值（仅作为 picker 的高亮项）：根据系统语言（`navigator.language` 起始为 `zh`）预选中文，否则预选英文

### UI 设计要点（与现有设计语言一致）

- **首次选择卡片**：复用辅助功能引导模态的尺寸 / 圆角 / 阴影；标题双语并列「选择语言 / Select language」；两个按钮等宽并排（中文 / English），点击即关闭
- **SideBar 切换器**：作为 footer 区第二个条目，视觉重量与「关于 TypeBridge」一致（小一号字号、灰阶配色）；点击触发 popover（不是原生 select），内容为两行可勾选项，宽度对齐 SideBar 容器；popover 出现位置为按钮右侧上浮，避免被窗口底边裁切

### 不在范围

- 多语言键盘布局适配
- 中英文混排正文的字体回退优化（直接使用现有 `system-ui` stack）
- 简体之外的中文变体（繁体 / 港台），后续如需扩展通过同一 dict 机制追加
- 官网（`website/`）、Go sidecar 的 i18n（这两处各有自己的 i18n 策略，不在桌面 App 本期范围）

### WebChat 移动端 SPA 跟随

桌面切换语言后，扫码打开的 WebChat 移动端页面也必须呈现对应语言：

- 桌面 Rust 在生成二维码 URL 时根据 `Settings.language` 注入 `&lang=zh|en`，未选择（空字符串）时不附加
- 移动端 SPA 自带 i18n 模块，按优先级 `URL ?lang=` → `localStorage tb_webchat_lang` → `navigator.language`（zh* → zh，否则 en）解析当前语言
- 进入页面后语言固定到页面销毁，不做响应式切换；下一次扫码时按上面优先级重算（用户曾访问过的 host 会复用 localStorage）
- 覆盖范围：握手卡片、错误屏、聊天页（header/状态/空态/输入框/按钮 aria）、消息状态、语音引导弹窗、PC 拦截页

### 9.6 适用场景文档内容

**定位**：让潜在用户理解 TypeBridge 的核心价值——不是"又一个聊天工具"，而是"IM → 桌面输入框"的桥接工具。通过具体场景描述，回答"我什么时候需要它"。

**文档结构**（章节锚点用于 Scroll Spy）：

1. **语音转文字，桌面直达** — 手机端用飞书/钉钉/企微内置的语音转文字功能说出内容，TypeBridge 自动将转写结果注入电脑端输入框。典型场景：写邮件、回复 Slack/Teams 消息、填写表单——不想打字时，说一句话就够了。
2. **AI Coding Agent 的语音搭档** — 在 Cursor、Copilot Chat、Windsurf 等 AI coding 工具中，用户通常需要手动在对话框中输入指令。配合 TypeBridge，可以在手机上用语音说出需求（"帮我重构这个函数，减少参数数量"），转写结果直接注入 AI Agent 的输入框，无需触碰键盘即可驱动 AI 执行任务。编码提效的全新范式。
3. **高频文档产出** — 对于需要持续产出文字内容的用户（撰写文档、起草报告、记录会议纪要），借助手机的语音转文字能力，一边说话一边让内容实时流入电脑文档——不用等说完再整理，说出来的就是文档草稿。配合「输入后自动提交」功能，语音产出流畅到连回车键都不用按。
4. **跨设备文本流转** — 在手机上复制的链接、地址、长文本，通过 IM 机器人发送即可直达电脑端当前输入框，无需"复制→邮件/云笔记→电脑再复制→粘贴"的繁琐链条。一句话发送，一步到位。
5. **团队协作提效** — 团队成员在飞书/钉钉群中 @ 机器人发送命令或数据（如 IP 地址、配置参数、SQL 片段），TypeBridge 自动注入到指定成员的电脑输入框。运维人员用手机发一条指令，代码编辑器里直接出现——告别"手机读→手动打字"的痛苦。
6. **行动号召** — "找到你的场景？下载 TypeBridge，5 分钟完成配置，立刻开始" + 下载链接

**文档页面底部**：引导到接入方式——"选定场景后，按以下教程完成你使用的 IM 渠道配置" + 四渠道链接（Web Chat/飞书/钉钉/企微，Web Chat 为官方推荐首选）。
