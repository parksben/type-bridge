# 产品定位 + 核心功能需求（§一 · §二 2.1–2.8）

---

## 一、产品定位

TypeBridge 是一个 macOS 桌面应用（基于 Tauri 构建），核心能力是：

**接收主流 IM 平台的机器人消息（飞书 / 钉钉 / 企业微信）→ 自动写入用户当前聚焦的输入框**

典型场景：用户在手机上通过 IM 向机器人发一段语音（转文字），桌面端自动将文字注入当前正在使用的编辑器、终端、浏览器输入框，实现"语音驱动桌面输入"。

**多渠道核心理念**：用户可以同时接入飞书 / 钉钉 / 企微的机器人，三家收到的消息**进入同一个 FIFO 队列**，依次粘到当前焦点输入框。各渠道彼此独立、互不阻塞——用户可以随手在任一 IM 给 bot 发消息，桌面侧串行处理。

---

## 二、功能需求

### 2.1 启动与配置窗口

- 首次启动弹出配置窗口，包含以下输入项：
  - 飞书应用 `App ID`（仅支持**自建应用**，飞书长连接不支持商店应用）
  - 飞书应用 `App Secret`
- 配置数据持久化到本地（`tauri-plugin-store`），二次启动自动复用，无需重新填写
- **关闭窗口即隐藏（不退出）**：点击窗口左上角红色关闭按钮，窗口隐藏，应用进程退入 Dock + 菜单栏托盘继续运行。再次点 Dock / 托盘图标即可秒级唤回，前端 React state 全部保留

**两个按钮（分阶段验证）：**

按钮分两步，对应飞书自建应用长连接接入的真实流程：

| 按钮 | 前置 | 行为 | 成功反馈 | 失败反馈 |
|------|------|------|---------|---------|
| **启动长连接** | App ID / App Secret 都已填写 | 先做表单校验（非空 + App ID 必须以 `cli_` 开头）→ 启动 Go sidecar 建立 WebSocket 长连接 | 状态变"已连接"+ 提示"请前往飞书开发者后台完成长连接验证" | 校验失败时在对应字段下显示红色错误文字；启动失败时在状态下方显示 sidecar stderr 关键行 |
| **测试连接** | 长连接必须处于"已连接" | 通过 Rust→Go stdin 发 `selftest` 命令，Go 侧并行跑 3 个**非破坏性 API probe** 校验消息链路 scope（下载图片 / 发表情 / 回复消息），逐项返回"通过 / 缺失 scope 列表" | 全部通过 → 清单四条全绿（凭据 + 3 项 scope）；否则按条目展示通过/缺失项 | 凭据错误（app_id / app_secret 非法）时清单整体标红，其他情况按失败条目单独提示去飞书开发者后台授权 |

**表单校验规则：**
- App ID 为空 → "App ID 不能为空"
- App ID 格式不对 → "App ID 应以 `cli_` 开头"
- App Secret 为空 → "App Secret 不能为空"
- 两个字段都合法才允许点击「启动长连接」

**连接测试清单显示：**

```
凭据可用（App ID / App Secret 能换到 token）        ✓
──────────────────────────────────
下载图片资源          im:message:readonly           ✓
发表情反应            im:message.reactions:write_only ✓
回复消息              im:message:send_as_bot        ✗ [去授权]
──────────────────────────────────
ⓘ 接收消息事件 需在飞书后台「事件配置」中单独完成
   ① 订阅方式：选择"使用长连接接收事件"并完成验证
   ② 添加事件：搜索 im.message.receive_v1 并勾选提交
   [去事件配置页 ↗]   [查看文档 ↗]
```

事件配置无法通过 API probe 自动校验。采用"完整列出步骤 + 直达应用事件配置页"的纯静态引导，不强求自动化。

**scope 校验的"最小粒度/兼容"规则：** 飞书同一个 API 通常存在多个可选 scope。probe 只关心 API 调用是否返回 `code=99991672`（Access denied）——用户勾了任意一个匹配 scope 都视为通过，不强制要求用最小粒度。

**设置项（同在配置窗口，状态变更立即持久化）：**

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| 输入后自动提交 | **开启** | 开启后，消息写入焦点输入框后模拟按下"提交按键"（默认 Enter），完成一键发送；关闭时仅注入文本不触发提交 |
| 提交按键 | `Enter` | "输入后自动提交"使用的按键。默认 Enter；支持自定义为任意按键或组合键（如 `⌘ + Enter`、`⌃ + Enter` 等）。仅当"输入后自动提交"开启时生效 |

### 2.2 飞书长连接

- 使用飞书开放平台 **WebSocket 长连接**模式接收机器人消息
  - 接口：`https://open.feishu.cn/open-apis/event/v1/im/message/receive_v1`（Event API v2）
  - 长连接端点：通过 `client.registerEventHandler` 注册消息事件
- 支持自动重连（断线后指数退避重试，最大间隔 60s）
- 连接状态实时写入日志

### 2.3 消息输入（剪贴板 + Cmd+V 策略）

- 收到飞书消息后，把内容写入系统剪贴板，然后**模拟 `Cmd+V`** 粘贴到当前前台应用
- 文本和图片走**同一条路径**：文本用 `NSPasteboardTypeString`，图片用 `NSPasteboardTypePNG`
- 粘贴执行完若设置里开启了"输入后自动提交"，追加模拟按下用户配置的提交按键
- 该方案**兼容性最好**——只要目标应用支持标准的"粘贴"行为，消息就能成功写入。特别是 Electron 应用（VSCode / Slack / Discord / Figma Desktop 等）通过 AX API 查询焦点不稳定，但粘贴完全没问题
- 权限要求：辅助功能（`AXIsProcessTrusted`），因为 `CGEventPost` 发全局按键事件需要该权限

**前台应用保护**（避免字符事件打回 TypeBridge 自己）：
- 粘贴前先通过 `NSWorkspace.frontmostApplication` 查当前前台应用的 bundle identifier
- 若前台是 TypeBridge 本身（`com.typebridge.app`），直接拒绝，`failure_reason = "当前前台是 TypeBridge 自己，请先切换到目标应用"`
- 这比之前用 AX 焦点元素 + role 白名单更可靠——不依赖目标应用是否暴露良好的 AX 树

**剪贴板策略取舍**：
- 粘贴后剪贴板保留消息内容（不回写旧值）。用户若想重新粘贴同一内容直接再按 Cmd+V 即可
- 图文混合消息：优先处理文本（剪贴板 + Cmd+V），再处理图片。图片处理会覆盖剪贴板

**辅助功能权限 —— 仍需授权（但作用已收窄）：**
- 应用启动完成后立刻通过 `AXIsProcessTrusted` 检查权限
- 新的剪贴板 + Cmd+V 策略下，**不再用** AX API 查焦点/读 AXRole；但 `CGEventPost` 发 `Cmd+V` 和自动提交按键给前台应用时，macOS TCC 仍要求应用在"辅助功能"列表里
- **未授予时展示为启动即弹的全屏模态（blocking gate）**：整个主窗口内容被模态覆盖，用户无法继续操作其他 tab。模态包含"为什么需要"说明 + 「前往系统设置」内联链接（深链到辅助功能面板）
- 模态底部两个等宽按钮：「已授权」（主按钮，点击主动校验权限）和「跳过」（次按钮，临时关闭模态）
- 授予期间后台每 3s 轮询一次 `check_accessibility`，用户在系统设置里打勾后模态自动消失

**历史设计（已废弃）**：v0.4 之前尝试过用 `AXUIElement` 查询焦点元素 + AXRole 白名单 + `CGEventPost` 逐字符模拟输入。该方案在 Electron / webview 类应用上 AX API 返回各种 AXError 导致消息无法写入，已整体替换为剪贴板 + Cmd+V 方案。

**输入后自动提交（默认开启）：**
- 注入完成后立即模拟按下"提交按键"，完成一键发送
- 默认按键为 `Enter`（macOS `kVK_Return`，keycode 0x24）
- 支持用户自定义：单键或组合键（`⌘/⇧/⌥/⌃` 与主键组合）
- 按键录入交互：点击"提交按键"字段进入捕捉模式 → 按下目标组合键 → 自动保存并退出；按 `Escape` 取消本次捕捉

**图片消息处理：**
- 飞书消息中含有图片时，Go sidecar 调用飞书 API 下载图片原始字节
- Rust 侧将图片写入 macOS **剪贴板**（`NSPasteboard`，`NSPasteboardTypeTIFF` / `NSPasteboardTypePNG`）
- 写入剪贴板后模拟 `Cmd+V` 触发粘贴，将图片粘入当前前台应用
- 若消息同时包含文本和图片：先注入文本，再粘贴图片（顺序保持消息原始顺序）
- 若目标输入框不支持图片粘贴，粘贴失败时写入日志，不影响文本注入

**不再做焦点校验**（v0.4.4+）：改用剪贴板 + Cmd+V 策略后，彻底不再查焦点——**能否真的粘到目标由用户自己负责**。唯一兜底：通过 PID 对比判断当前前台是不是 TypeBridge 自己，是就拒绝；否则粘贴命令照发。

### 2.4 应用生命周期 / Dock + 托盘交互

应用同时在 Dock 和菜单栏托盘上展示入口（v0.7.x 起）。两者都只是"快捷入口"——单击直接打开主窗口、抢焦点。**托盘不挂下拉菜单**（参考微信桌面端的体验，避免冗余的"打开主窗口"菜单项）。

| 触发 | 行为 |
|------|------|
| **窗口左上角红色关闭按钮** | 拦截 close 事件，**隐藏窗口**（保留前端 React state）。应用进程不退出，Dock 和托盘图标都继续可见 |
| **Dock 图标单击** | 通过 `RunEvent::Reopen` 检测 `has_visible_windows == false` 时调 `show_or_create_main_window`，让窗口重新可见并抢焦点 |
| **托盘图标单击** | 等价 Dock click，调同一个 `show_or_create_main_window`。托盘**不再有下拉菜单** |
| **Cmd+Q** | macOS 标准退出快捷键，正常退出进程 |
| **Dock 图标右键 → 退出** | macOS 标准退出路径，正常退出进程 |

**为什么去掉托盘下拉菜单**：原版菜单只有"打开主窗口"和"退出应用"两项，前者已经被单击行为覆盖，后者通过 Cmd+Q / Dock 右键退出已经能完成；保留菜单只是徒增一次右键交互成本，不带来真实的功能。

**托盘图标设计**：橙红渐变圆角底 + 白色桥拱图案，22pt×22pt 视觉面积适配 macOS 菜单栏标准高度。

### 2.5 主窗口（左侧侧边栏 + 内容区）

应用仅保留一个主窗口（820×560 px）。左侧 ~150px 竖向侧边栏，**所有 tab 平铺在顶部**：

| Tab | 内容区 | 备注 |
|-----|--------|------|
| **连接 TypeBridge** | 顶部一句话说明 + 横向子 tab 切换（WebChat / 飞书 / 钉钉 / 企微）+ 当前渠道的配置表单 | 原 v0.5 的"服务配置"板块 + 3 个独立 tab，v0.6+ 合并为单一 sidebar tab + 内容页顶部横向 tab |
| **输入设置** | 输入后自动提交 + 提交按键 | |
| **历史消息** | chip 筛选（全部 / 飞书 / 钉钉 / 企微）+ 消息卡片列表 | |
| **系统日志** | 按时间倒序列表（含 `[渠道]` 彩色前缀） | 不含消息 / 注入事件（那些归历史消息 tab） |
| **关于 TypeBridge** | logo + 名称 + 版本号 + 检查更新按钮 | v0.7.x 新增；详见 §2.11 |

**连接状态展示（v0.7+）**：**不再**在侧边栏底部单独列每渠道的连接状态行。连接状态直接体现在「连接 TypeBridge」tab 内部的横向子 tab 上——每个子 tab 在连接成功时左侧会有一个绿色小圆点。

#### Tab 1: 连接 TypeBridge

内容区自上而下三部分：

**① 一句话说明**（顶部贯穿横条）：「通过内置 WebChat 或 IM 应用机器人连接 TypeBridge 进行输入」。

**② 横向子 tab**（飞书 | 钉钉 | 企微）：点击切换下方的渠道配置面板；active tab 用 2px accent 色下划线 + 文字加深。

**③ 渠道配置面板**（占据剩余高度，独立滚动）：
- 飞书：顶部「还没有自建应用？」引导 banner → App ID / App Secret 输入 → 启动长连接 + 测试连接 → scope 检查清单
- 钉钉：结构对齐，字段名改 Client ID / Secret
- 企微：v0.6 暂占位，P3 接入智能机器人长连接后落地

#### Tab 2: 历史消息

展示所有收到并处理过的消息，按进入时间**倒序**排列（最新在最上方）。

**消息卡片**包含：
- 收到时间（相对时间：刚刚 / 2 分钟前 / 昨天 15:30 / 2026-04-30）
- 消息正文（文本 + 图片缩略图）
- 状态 tag：`已入队` / `处理中` / `已发送` / `失败`
- **失败原因分两个独立层级展示**：
  1. **本地输入失败**（橙色小字，紧贴消息正文）：消息写入这一步失败。常见原因：`辅助功能权限未授予`、`当前前台是 TypeBridge 自己`、`剪贴板写入失败`
  2. **飞书反馈被拒**（红色 banner）：机器人给这条消息发表情 / thread 回复时被飞书拒绝（scope 权限不足、频控、emoji_type 非法等）
- 两类错误可以**同时出现**
- 卡片右侧操作按钮：**复制**（文本 / 图片到系统剪贴板）、**删除**（从历史中移除该消息）

> 设计取舍：v0.4 原本有"重发"按钮，但实测点击按钮本身会把焦点夺走 → 消息到达时又变成无焦点，一键死循环。所以把"重发"替换为"复制"。

**顶部"清空"按钮（硬清空 + 取消队列）**：
- 点击后弹出**应用内确认弹窗**（不是浏览器原生 `window.confirm`，因为 Tauri WKWebView 默认不实现该 panel，调用直接返回 false）
- 确认后会：清空 `~/.typebridge/history.json`、清空 UI 卡片，并把队列中尚未注入的对应条目标记为取消

**持久化**：
- 消息历史写入 `~/.typebridge/history.json`（应用关闭后仍保留）
- 最多保留 **500 条**，超出自动滑动淘汰最早的（`FIFO`）
- 图片内容不内嵌，存到 `~/.typebridge/images/<message_id>.<ext>`，历史记录只存相对路径

#### Tab 5: 系统日志

展示应用连通性 / 错误 / 权限 / 启动事件等运维日志（**不含用户消息内容**，那些归"历史消息"tab）。保留"在访达中显示"按钮跳转到日志文件目录。

**顶部"清空"按钮（软清空）**：
- 仅清理当前 UI 中可见的日志条目；`~/Library/Logs/TypeBridge/*.log` 文件 **不动**
- 清空后新产生的事件仍正常展示；文件日志继续按天滚动写入

#### Tab 6: 输入设置

控制消息到达本地后的输入行为，v0.5 从「连接」tab 拆出，独立于飞书凭据配置。

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| **输入后自动提交** | 开启 | 开启后，消息写入焦点输入框后模拟按下"提交按键"（默认 Enter），完成一键发送；关闭时仅注入文本不触发提交 |
| **提交按键** | `Enter` | "输入后自动提交"使用的按键。默认 Enter；支持自定义为任意按键或组合键（如 `⌘ + Enter`、`⌃ + Enter` 等）。仅当"输入后自动提交"开启时生效 |

### 2.6 飞书消息双向回复

机器人在接收与处理消息的各阶段，向用户给出可视反馈：

| 阶段 | 回复形式 | 具体内容 |
|------|---------|---------|
| 收到消息（入队） | 表情反应 | 给原消息加 `EYES`（👀，"已看到"） |
| 处理成功（已注入） | 表情反应 | 将原有 `EYES` 替换为 `DONE`（✅） |
| 处理失败 | 表情反应 + 文字 | 将原有 `EYES` 替换为 `SAD`（❌），并在原消息 thread 下回复一条文字消息说明失败原因 |
| 处理中（长任务或等待确认） | 表情反应 | 保持 `EYES`，直到进入终态 |

**设计取舍：**
- 成功场景不额外发文字，避免污染群聊；失败场景必须发文字，方便用户明白为什么
- 表情反应失败（权限/网络问题）不影响主流程注入，仅记录到系统日志

### 2.7 消息队列机制

用户高频发消息时必须**严格串行**处理，避免多条消息同时注入打架：

- 所有收到的消息进入**单一 FIFO 队列**
- 同一时刻只有一条消息处于 `处理中`，其余 `已入队` 排队等待
- 处理完成（成功或失败）后立即出队并取下一条
- UI 中每条消息的状态 tag 实时反映当前所处阶段

**状态机：**

```
接收消息 → [已入队] ──worker 取出──► [处理中] ──┬─► [已发送]
                                              │
                                              └─► [失败]
```

**失败重试**：不自动重试。失败后消息保持 `失败` 状态，用户可在消息历史 tab 中点"重发"重新入队。

**并发硬约束**：严格单 worker 串行；注入期间新消息只入队不注入。

### 2.8 日志系统

**日志窗口内容（实时滚动）：**
- 飞书连通性日志（连接成功、断线、重连尝试）
- 收到的消息内容（发送人、时间、消息文本，敏感内容可选脱敏）
- 注入状态（目标应用名、注入成功/失败原因）
- 权限状态变化

**日志文件：**
- 写入路径：`~/Library/Logs/TypeBridge/typebridge-YYYY-MM-DD.log`
- 按天滚动，保留最近 30 天
- 日志窗口底部提供「在访达中显示」按钮，点击后用 `NSWorkspace.selectFile` 在 Finder 中高亮该文件
- **多渠道日志前缀**：每条日志带 `[飞书]` / `[钉钉]` / `[企微]` 渠道前缀；与渠道无关的全局事件（启动、辅助功能权限等）不带前缀
